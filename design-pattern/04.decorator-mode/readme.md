### 装饰模式

- 将 要被装饰的类 物（人） 做为最根本的基类
- 装饰类（decorator类） 继承 要被装饰的类
- 实例化多个装饰子类
- 实例化一个 要被装饰的类 物人类

- 使用的时候  先指定人 然后层层装饰即可


### 名言

    装饰模式是利用 setComponent() - 在decorator类中定义的，  来对对象进行包装的，这样每个修饰对象的实现就和如何使用这个对象分离开了，每个修饰对象只关心自己的功能，不需要关心如何被增加到对象链中。

    如果只有一个concreteComponent类 而没有抽象的 component类，那么decorator类可以是ConreteComponent的一个子类。同样道理，如果只有一个concreteDecorator类，那么就没有必要建立一个单独的decorator类，而可以把decorator 和 concreteDecorator的责任合并成一个类。

### 总结
    装饰模式是为己有功能动态地添加 更多功能的一种方式。

    使用场景：
        当系统需要新的功能的时候，是向旧的类中添加新的代码，这些新加的代码通常装饰了原有类的核心职责和主要行为。

        在主类中加入了新的字段，新的方法和新的逻辑，从而增加了主类的复杂度，就像你起初的那个人类，而装饰模式却提供了一个非常好的解决方案。它把每个要装饰的功能放在单独的类中，并让这个类包装他所要修饰的对象，因此，当需要执行特殊行为时，客户代码就可以在运行时根据需要有选择地，按顺序地使用装饰功能包装对象了。

#### 优点
    把类中的装饰功能从类中搬移去除，这样可以简化原有的类

    这样做更大的好处就是有效地把类的核心职责和装饰功能区分开了，而且可以去除相关类中重复的装饰逻辑

    要保证装饰类之间彼此独立，这样他们就可以任意的顺序进行组合了
